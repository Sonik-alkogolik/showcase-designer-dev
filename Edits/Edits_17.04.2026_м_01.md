# План: защита лимитов тарифа при создании и импорте товаров

Обновлено: 2026-04-16

## Контекст проблемы

Сейчас в системе есть два тарифных плана:

### Бесплатный
- 1 магазин
- до 20 товаров
- Telegram Web App

### Платный
- 5 магазинов
- до 200 товаров
- импорт товаров из Excel
- приоритетная поддержка

На практике сейчас наблюдается такая проблема:

- во фронте на бесплатном тарифе корректно отображается лимит в 20 товаров;
- но при импорте в базу может загрузиться намного больше товаров;
- это означает, что текущая защита работает только частично;
- ограничение не останавливает сам процесс создания записей внутри импорта.

---

## Главная причина

Скорее всего, текущая проверка стоит только на входе в контроллер:

- если товаров сейчас меньше лимита, импорт разрешается;
- дальше импортёр построчно создаёт товары;
- итоговое количество после импорта уже никто не ограничивает.

Пример:

- лимит тарифа = 20
- в магазине уже 18 товаров
- пользователь загружает файл на 100 строк
- контроллер видит, что `18 < 20`, и разрешает импорт
- импортёр создаёт все 100 товаров
- база переполняется

Итог:
- UI может показывать часть товаров или лимит,
- но база уже загрязнена лишними записями.

---

## Цель

Нужно сделать так, чтобы:

1. лимит тарифа проверялся не только во фронте;
2. лимит тарифа проверялся не только в контроллере до запуска импорта;
3. сам импорт не мог создать товаров больше разрешённого количества;
4. доступность функции импорта зависела от тарифа;
5. пользователь получал понятный ответ, почему импорт ограничен или запрещён.

---

## Бизнес-логика, которую нужно зафиксировать

## 1. Ограничение по количеству товаров

У каждого тарифа есть значение лимита товаров.

Например:
- бесплатный = 20
- платный = 200

Дальше система должна считать:

- `limit = значение тарифа`
- `currentCount = текущее количество товаров в магазине`
- `availableSlots = limit - currentCount`

Правила:
- если `availableSlots <= 0`, новые товары создавать нельзя;
- если `availableSlots > 0`, можно добавить только это количество.

---

## 2. Ограничение по доступности Excel-импорта

Нужно отдельно определить:
- доступен ли импорт из Excel на конкретном тарифе.

Варианты:
- бесплатный тариф: импорт запрещён;
- платный тариф: импорт разрешён.

Это важно, потому что:
- лимит товаров и доступ к функции импорта — это не одно и то же;
- даже если лимит ещё не исчерпан, сам Excel-импорт может быть недоступен по тарифу.

---

## 3. Разделение правил для одиночного создания и массового импорта

### Обычное создание одного товара
Проверка в контроллере обычно достаточна:
- если лимит достигнут, вернуть ошибку;
- иначе создать товар.

### Массовый импорт
Проверки в контроллере недостаточно:
- потому что импорт идёт построчно;
- нужно ограничение внутри самого импортёра.

---

## Архитектурное решение

Нужно ввести защиту на нескольких уровнях.

---

## Уровень 1. Фронт

Фронт нужен для UX, но не для безопасности.

Что он должен делать:

### Для бесплатного тарифа
- либо скрывать кнопку "Импорт из Excel";
- либо показывать её disabled;
- либо при открытии модалки сразу выводить сообщение:
- "Импорт из Excel доступен только на платном тарифе"

### Для платного тарифа
- разрешать открывать модалку;
- можно показывать оставшийся лимит:
- "Можно добавить ещё 37 товаров"

### Важно
Фронт не должен быть единственной защитой.
Даже если кнопка скрыта, бэкенд всё равно обязан проверять тариф и лимит.

---

## Уровень 2. Контроллер импорта

Контроллер должен делать предварительные проверки до запуска импорта.

### Что должен проверять `ImportController::import()`

#### 1. Право на Excel-импорт
Проверить, разрешён ли импорт на текущем тарифе.

Если нет:
- вернуть `403`
- сообщение вроде:
- "Импорт из Excel доступен только на платном тарифе"

#### 2. Лимит товаров
Вычислить:
- текущий лимит тарифа;
- текущее количество товаров в магазине;
- остаток свободных слотов.

Если остаток `<= 0`:
- вернуть `403`
- сообщение:
- "Вы достигли лимита товаров по вашему тарифу"

#### 3. Передача лимита в импортёр
Если импорт разрешён, контроллер должен передать в `AdvancedProductsImport`:
- лимит тарифа;
- текущее количество товаров;
- доступный остаток.

Именно остаток должен стать реальным ограничителем внутри импорта.

---

## Уровень 3. Импортёр `AdvancedProductsImport`

Это главный слой защиты.

### Почему это важно
Только импортёр знает, сколько строк уже реально обработано и сколько товаров уже реально создано.

### Что он должен делать
Импортёр должен хранить:
- максимально разрешённое количество новых товаров в рамках текущего запуска;
- сколько уже успешно импортировано;
- сколько строк было пропущено из-за лимита.

### Что должно происходить в `model()`
Перед созданием каждого нового товара:

- если лимит на новые товары уже исчерпан:
- не создавать товар;
- либо возвращать `null`;
- либо фиксировать отдельную ошибку/статус "пропущено из-за лимита".

Именно так база не сможет превысить тарифный лимит.

---

## Варианты поведения при превышении лимита

Есть два рабочих сценария.

---

## Вариант A. Частичный импорт до лимита

Пример:
- лимит = 20
- уже есть 18 товаров
- осталось 2 места
- пользователь загружает 50 строк

Поведение:
- импортируются только первые 2 валидных товара;
- остальные строки не создаются;
- в ответе возвращается информация:
- импортировано: 2
- пропущено из-за лимита: 48

### Плюсы
- пользователь получает хотя бы частичный результат;
- удобно при больших файлах.

### Минусы
- результат не всегда очевиден, если пользователь ожидал импорт всего файла.

---

## Вариант B. Полный отказ, если файл больше остатка

Пример:
- можно добавить только 2 товара
- файл содержит 50 строк

Поведение:
- импорт вообще не запускается;
- пользователь получает ошибку:
- "По вашему тарифу можно добавить ещё только 2 товара"

### Плюсы
- поведение максимально прозрачное;
- нет "частичного" результата.

### Минусы
- менее удобно для пользователя.

---

## Рекомендуемый вариант

Для текущего проекта логичнее такой подход:

### Бесплатный тариф
- импорт Excel полностью запрещён.

### Платный тариф
- импорт разрешён;
- но количество новых товаров всё равно ограничено остатком по тарифу.

### При превышении лимита
Предпочтительно использовать **частичный импорт** с понятным отчётом:
- сколько импортировано;
- сколько пропущено;
- почему пропущено.

Это даст лучший UX.

---

## Что нужно вернуть в API-ответе

После импорта полезно возвращать не только `success_count`, но и дополнительную информацию.

Например:

- `limit`
- `current_count_before_import`
- `available_slots_before_import`
- `imported_count`
- `skipped_due_to_limit`
- `total_rows`

### Пример смысла ответа
- лимит: 20
- было товаров: 18
- доступно слотов: 2
- импортировано: 2
- пропущено из-за лимита: 48

Это поможет и фронту, и диагностике.

---

## Что нужно проверить в существующей системе

Перед реализацией завтра нужно быстро проверить следующие вещи.

### 1. Где хранится лимит тарифа
Нужно найти:
- откуда берётся `getProductsLimit()`;
- где у тарифа хранится `value`;
- есть ли отдельный флаг "доступен Excel-импорт".

### 2. Как сейчас определяется платный/бесплатный тариф
Нужно понять:
- по `plan`;
- по `value`;
- по набору features;
- по активной подписке.

### 3. Как правильно определить доступ к импорту
Нужно решить:
- делать через `plan === 'paid'`;
- делать через отдельный флаг `can_import_excel`;
- делать через список возможностей тарифа.

Рекомендуется отдельный capability/feature, а не жёсткая проверка по названию плана.

---

## Предпочтительная модель правил

Лучше мыслить не названиями тарифов, а возможностями.

### Пример
У тарифа есть:
- `products_limit`
- `shops_limit`
- `can_import_excel`

Тогда логика становится стабильной:
- лимит товаров = `products_limit`
- доступ к Excel = `can_import_excel`

Это лучше, чем проверять строки типа:
- `if plan == free`
- `if plan == paid`

---

## План работ на завтра

## Этап 1. Аудит текущей логики тарифа
Проверить:
- где живёт `getProductsLimit()`;
- как определяется активная подписка;
- где хранится значение лимита;
- можно ли быстро получить флаг доступа к Excel.

### Результат этапа
Понять, откуда брать:
- лимит товаров;
- право на Excel-импорт.

---

## Этап 2. Защита в `ImportController`
Добавить логику:

### Проверка 1
Если тариф не поддерживает Excel:
- вернуть ошибку `403`.

### Проверка 2
Посчитать:
- `currentCount`
- `limit`
- `availableSlots`

Если `availableSlots <= 0`:
- вернуть ошибку.

### Проверка 3
Передать `availableSlots` в `AdvancedProductsImport`.

### Результат этапа
Импорт не стартует, если он вообще не разрешён.

---

## Этап 3. Защита в `AdvancedProductsImport`
Добавить состояние:

- `availableSlots`
- `importedWithinLimit`
- `skippedDueToLimit`

Внутри `model()`:
- если `importedWithinLimit >= availableSlots`, не создавать новые товары;
- увеличить счётчик пропущенных по лимиту.

### Результат этапа
Даже если контроллер пропустил импорт, база не переполнится.

---

## Этап 4. Улучшение ответа после импорта
Добавить в JSON-ответ:
- сколько было разрешено;
- сколько импортировано;
- сколько пропущено из-за лимита.

### Результат этапа
Фронт сможет показать пользователю понятный итог.

---

## Этап 5. Улучшение фронта
Опционально:

### Для бесплатного тарифа
- скрыть кнопку импорта;
- или disabled + tooltip;
- или модалка с upsell.

### Для платного
- показывать остаток слотов перед импортом.

### Результат этапа
Пользователь заранее понимает ограничения.

---

## Сценарии тестирования

После реализации нужно проверить минимум следующие кейсы.

### Кейс 1. Бесплатный тариф, 0 товаров
Ожидание:
- импорт Excel запрещён полностью.

### Кейс 2. Бесплатный тариф, 20 товаров
Ожидание:
- создать товар нельзя;
- импортировать нельзя.

### Кейс 3. Платный тариф, 0 товаров
Ожидание:
- импорт доступен;
- товары создаются до лимита.

### Кейс 4. Платный тариф, 198 товаров, файл на 10 строк
Ожидание:
- импортируются только 2;
- остальные строки пропускаются по лимиту.

### Кейс 5. Платный тариф, уже 200 товаров
Ожидание:
- импорт сразу запрещён.

### Кейс 6. Платный тариф, файл содержит битые строки
Ожидание:
- валидные строки импортируются;
- лимит не превышается;
- ошибки и лимит считаются отдельно.

## Что важно не забыть

### 1. Фронт не является защитой
Даже если кнопка скрыта, бэкенд всё равно обязан всё проверить.

### 2. Контроллер не является достаточной защитой
Для массового импорта лимит обязательно должен контролироваться внутри импортёра.

### 3. Нужно разделять:
- право использовать функцию импорта;
- количественный лимит товаров.

### 4. Ответ после импорта должен быть прозрачным
Чтобы пользователь понимал:
- что импортировалось;
- что не импортировалось;
- почему.

---

## Итоговое целевое поведение

Система должна работать так:

### Бесплатный тариф
- можно иметь до 20 товаров;
- Excel-импорт недоступен;
- при попытке импорта пользователь получает понятный отказ.

### Платный тариф
- можно иметь до 200 товаров;
- Excel-импорт доступен;
- импорт не может создать товаров больше, чем разрешено;
- пользователь видит понятный результат.

---

## Короткий вывод

Сейчас лимит работает только частично, потому что контролируется слишком рано.

Нужно перенести реальный контроль в два места:
1. в `ImportController` — предварительная проверка доступа и остатка;
2. в `AdvancedProductsImport` — жёсткая остановка создания товаров по достижении лимита.

Тогда:
- база перестанет переполняться;
- тарифные ограничения станут настоящими, а не только визуальными;
- поведение системы будет предсказуемым.
