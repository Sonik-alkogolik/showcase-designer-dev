# Авто-восстановление сессии при повторном заходе (без hard refresh)

Обновлено: 2026-04-19  
Файл правок: `Edits_19.04.2026_м_01.md`

## Проблема

При открытии старой вкладки на следующий день пользователь мог видеть серию `500` по API (`/api/profile`, `/api/shops`, `/api/subscription/plans`) и исправлял это только через hard refresh с очисткой кэша.

## Решение

В `useAuth` добавлен глобальный `axios` response interceptor:

1. Если есть активный токен и пришёл `401/419`:
- токен сбрасывается;
- пользователь перекидывается на `/login?reason=session_expired`.

2. Если есть токен и пришёл `500` на критичных bootstrap-endpoint:
- `/api/profile`
- `/api/shops`
- `/api/subscription/plans`
- `/api/user`

выполняется тот же сценарий безопасной переавторизации:
- сброс токена;
- редирект на `/login`.

3. Дополнительно в `loadProfile()` при `500` токен также очищается как fallback.

## Эффект

Пользователь больше не остаётся на “поломанной” старой вкладке с бесконечными 500.  
Система мягко переводит его в повторный вход, что лучше для UX и предсказуемости.
