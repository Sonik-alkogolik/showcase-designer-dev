# SBP PAYMENTS ROLLOUT PLAN

Обновлено: 2026-04-22

## Цель

Подключить реальную онлайн-оплату через СБП (QR) для двух контуров:

1. Оплата заказа покупателем (Telegram Mini App).
2. Оплата тарифа `business` владельцем магазина.

## Текущее состояние (as-is)

1. По заказам уже есть каркас оплаты через ЮKassa.
2. По подпискам `subscribe` активирует `business` сразу, без фактического платежа.
3. Значит ключевая доработка — перевести активацию подписки на модель `pending -> paid -> active`.

## Базовое решение

1. Используем одного провайдера (ЮKassa) с методом оплаты СБП.
2. QR-код/платёжная страница генерируется провайдером.
3. Источник истины по успешной оплате: webhook + server-side верификация статуса платежа.

## Единая модель статусов

### Заказы

1. `draft`
2. `pending_payment`
3. `paid`
4. `cancelled` / `expired`

### Подписки

1. `inactive`
2. `pending_payment`
3. `active`
4. `expired` / `cancelled`

## Поток оплаты СБП (QR)

1. Клиент выбирает оплату онлайн.
2. Backend создаёт платеж у провайдера с методом СБП.
3. Клиент получает `confirmation_url` и показывает QR / страницу оплаты.
4. Пользователь оплачивает в банковском приложении.
5. Провайдер отправляет webhook.
6. Backend:
   - проверяет webhook и статус у провайдера;
   - применяет idempotent обновление сущности;
   - переводит заказ/подписку в финальный статус.

## Контур 1. Заказы

1. В checkout добавить явный выбор:
   - `Оплатить онлайн (СБП)`,
   - `Оплатить менеджеру`.
2. При онлайн-оплате создавать заказ в `pending_payment`.
3. После `payment.succeeded` переводить в `paid`.
4. При `canceled/expired` давать безопасный повтор оплаты без дублей заказа.

## Контур 2. Подписки business

1. `POST /subscription/subscribe` не должен сразу создавать `active`.
2. Вместо этого:
   - создаётся платеж подписки (`pending_payment`);
   - после `payment.succeeded` создаётся/активируется подписка `active`.
3. До успешной оплаты возможности `business` не включаются.

## Безопасность (обязательный gate)

1. Проверка webhook (секрет/подпись).
2. Idempotency:
   - создание платежа,
   - обработка webhook,
   - повторная доставка событий.
3. Безопасный polling статуса оплаты (без возможности перебора `payment_id`).
4. Серверный пересчёт суммы заказа и суммы тарифа.
5. Rate limit для payment/webhook/auth маршрутов.

## Наблюдаемость и операции

1. Логировать:
   - создание платежа,
   - входящий webhook,
   - результат reconcile.
2. Метрики:
   - conversion в `paid`,
   - доля `canceled/expired`,
   - latency webhook.
3. Reconcile job:
   - периодически проверяет `pending_payment`,
   - подтягивает статус у провайдера.

## Поэтапный запуск

### Этап A (dev/stage)

1. Включить СБП flow в тестовом окружении.
2. Прогнать сценарии:
   - success,
   - cancel,
   - timeout,
   - duplicate webhook.

### Этап B (limited prod)

1. Включить для ограниченной группы магазинов.
2. Наблюдать 3-5 дней.
3. Фиксировать инциденты и улучшать runbook.

### Этап C (full prod)

1. Включить для всех пользователей.
2. Сохранить kill-switch на случай деградации провайдера.

## Go / No-Go Checklist

1. Подписка `business` не активируется без успешного платежа.
2. Заказ с онлайн-оплатой корректно проходит в `paid` через webhook.
3. Дубли webhook не ломают состояние.
4. Polling статуса защищён от перебора.
5. Reconcile job закрывает зависшие `pending_payment`.
6. Подготовлен rollback в `manual/draft` режим.

## Rollback

1. Отключить онлайн-оплату feature-flag/конфигом.
2. Оставить заказы в ручном подтверждении.
3. Для подписок временно блокировать self-service upgrade до стабилизации.
4. Провести пост-мортем и повторный rollout по этапам.

## Definition of Done

1. Оба платежных контура работают на production с подтверждением через webhook.
2. Критичные security-пункты по платежам закрыты.
3. Есть runbook для поддержки и on-call.
4. Подтверждена стабильность limited rollout перед full launch.
