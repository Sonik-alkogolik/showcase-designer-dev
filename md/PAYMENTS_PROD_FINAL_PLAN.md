# PAYMENTS PROD FINAL PLAN

Обновлено: 2026-04-22

## Цель

Закрыть финальный этап запуска оплаты на проде по двум независимым контурам:

1. Оплата заказа покупателем в Telegram Mini App.
2. Оплата тарифа `business` в личном кабинете владельца.

## Принцип запуска

1. Контуры запускаются отдельно, чтобы не смешивать риски.
2. Любая успешная оплата подтверждается только через webhook + верификацию у провайдера.
3. При проблемах с онлайн-оплатой сервис остаётся работоспособен через fallback (manual/draft).

## Этап 1. Продуктовые правила

### Заказы

1. Статусы: `draft -> pending_payment -> paid -> cancelled/expired`.
2. Срок жизни неоплаченного платежа: `15-30 минут`.
3. Повторная попытка оплаты не должна создавать дубли заказов.

### Подписки

1. Статусы: `inactive -> pending_payment -> active -> expired/cancelled`.
2. Тариф `business` не активируется до подтверждённой оплаты.
3. Автопродление включается только при явном согласии пользователя.

## Этап 2. Безопасность и устойчивость (обязательный gate)

1. Верификация webhook ЮKassa (подпись/секрет).
2. Idempotency:
   - создание платежа;
   - обработка webhook;
   - повторные уведомления.
3. Безопасный polling статуса оплаты (без возможности перебора `payment_id`).
4. Серверный пересчёт суммы (не доверяем суммам с фронта).
5. Rate limit для чувствительных endpoints:
   - создание платежа,
   - webhook,
   - auth endpoints.

## Этап 3. UX оплаты заказа (Mini App)

1. В checkout явно доступны два сценария:
   - `Оплатить онлайн`,
   - `Оплатить менеджеру`.
2. После онлайн-оплаты:
   - редирект в провайдер,
   - экран ожидания подтверждения.
3. При успехе:
   - заказ получает `paid`,
   - пользователь видит подтверждение.
4. При неуспехе/таймауте:
   - понятное сообщение,
   - заказ не теряется.

## Этап 4. UX оплаты тарифа business

1. При выборе `business` создаётся сущность оплаты подписки (`pending_payment`).
2. После `payment.succeeded`:
   - подписка переводится в `active`,
   - проставляется корректный `expires_at`.
3. При `payment.canceled/expired`:
   - подписка не активируется,
   - пользователь может безопасно переинициировать оплату.

## Этап 5. Soft Launch и наблюдаемость

1. Запуск на ограниченной группе магазинов.
2. Ежедневный мониторинг:
   - conversion в успешную оплату,
   - доля ошибок оплаты,
   - задержка webhook.
3. Kill-switch:
   - быстрый возврат к режиму `draft/manual`,
   - без остановки базового core-flow.

## Definition Of Done (финальный этап PROD)

1. Оба контура оплаты проходят E2E на production.
2. Критичные риски по webhook/payment из `md/SECURITY_DEBT_TODO.md` закрыты.
3. Есть runbook инцидента:
   - "оплата у провайдера успешна, статус в системе не обновился".
4. Есть ежедневная сверка оплат (операционный отчёт).
5. Поддержка знает регламент ответов пользователю по всем исходам оплаты.

## Release Checklist (перед включением всем)

1. Пройден тест-пакет:
   - success payment,
   - canceled payment,
   - timeout/expired payment,
   - duplicate webhook delivery.
2. Проверены алерты и логи на проде.
3. Проверен fallback в manual/draft.
4. Обновлены runbook и FAQ для поддержки.
5. Назначен owner на первые 72 часа после включения.

## Rollback Plan

1. Отключить онлайн-оплату feature-flag/конфигом.
2. Оставить создание заказа и обработку вручную.
3. Сохранить платёжные артефакты для последующей сверки.
4. Провести пост-мортем и повторный запуск только после устранения причины.
